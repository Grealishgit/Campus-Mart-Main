# Backend Project Structure Guide

This document is an instruction manual for working inside `backend/`.  
It is written for humans and LLMs that need to understand the backend quickly, safely, and accurately.

## What this backend is

`backend/` is a **Node.js + Express + PostgreSQL** API for the Campus Mart platform.

### Core runtime assumptions

- **Module system:** CommonJS (`"type": "commonjs"`)
- **Entry point:** `server.js`
- **Database:** PostgreSQL
- **API style:** REST endpoints under `/api/...`
- **Validation:** route-specific validator modules
- **Security:** auth middleware, rate limiting, sanitization, helmet, JWT
- **Uploads:** handled through `multer` and Cloudinary-related helpers
- **Testing:** Jest + Supertest

## Package scripts

From `backend/package.json`:

- `npm start` → `node server.js`
- `npm run server` → `nodemon server.js`
- `npm run setup-db` → `node setup-db.js`
- `npm test` → `jest --detectOpenHandles --forceExit`
- `npm run test:watch` → `jest --watch`
- `npm run test:coverage` → `jest --coverage`

### Important note

The backend `README.md` mentions `npm run dev`, but that script does **not** exist in `package.json`.  
Use `npm run server` for development unless the script list is updated later.

## How the backend is organized

The backend follows a classic layered structure:

1. **Routes** receive requests
2. **Validators** check request shape and constraints
3. **Middleware** handles auth, rate limits, uploads, and request sanitation
4. **Controllers** implement business logic
5. **Config** holds DB and schema setup
6. **Utils** provide reusable helpers
7. **SQL files** define and seed database state

When editing or extending the backend, keep this flow intact.

### `backend/config/`

- `config/db.js`  
  PostgreSQL connection/configuration module.

- `config/original_schema.sql`  
  Original schema snapshot. Useful as a reference when comparing schema changes.

- `config/schema.sql`  
  Current canonical schema file. Use this when recreating or updating the database structure.

#### Rules for `config/`

- If database columns, relationships, or table definitions change, update `schema.sql`.
- If there is migration-like logic in `setup-db.js`, keep it aligned with the schema files.

- `controllers/adminController.js`  
  Admin feature logic.

- `controllers/authController.js`  
  Authentication and account-related logic.

- `controllers/chatController.js`  
  Chat and conversation logic.

- `controllers/favoritesController.js`  
  Favorites/saved listings logic.

- `controllers/leaseController.js`  
  Lease-related listing logic.

- `controllers/listingController.js`  
  Listing CRUD and filtering logic.

- `controllers/orderController.js`  
  Order and purchase workflow logic.

- `controllers/saleController.js`  
  Sale-related listing logic.

#### Rules for `controllers/`

- Keep request parsing in routes; keep business logic here.
- Controllers should not own URL definitions.
- Controllers should stay consistent with the validators and middleware that call them.

### `backend/lib/`

This folder contains SQL dump and seed-style files.

- `lib/campus_mart_dump_2026-04-23T21-23-13-364Z.sql`  
  Timestamped database dump/backup artifact.

- `lib/listings.sql`  
  Listing seed or SQL reference file.

- `lib/User.sql`  
  User-related SQL reference or seed file.

#### Rules for `lib/`

- Treat this folder as SQL support material.
- Do not assume it is application code.
- Verify whether a file is a seed, backup, or schema reference before changing it.

---

### `backend/middleware/`

Middleware applies reusable request handling before controllers run.

- `middleware/authMiddleware.js`  
  Authentication/authorization checks.

- `middleware/rateLimiter.js`  
  Request throttling middleware.

- `middleware/uploadMiddleware.js`  
  File upload handling for multipart requests.

- `middleware/validateRequest.js`  
  Request validation wrapper / helper.

#### Rules for `middleware/`

- Use middleware when the same concern must be shared across multiple routes.
- Keep middleware small and focused.
- If a route needs auth, upload handling, or validation, compose it here rather than duplicating logic in controllers.

---

### `backend/routes/`

Routes define the API surface. They map HTTP methods and paths to middleware and controllers.

- `routes/adminRoutes.js`  
  Admin API routes.

- `routes/authRoutes.js`  
  Authentication routes.

- `routes/chatRoutes.js`  
  Chat API routes.

- `routes/favouriteRoutes.js`  
  Favorites routes. Note the British spelling in the filename: `favouriteRoutes.js`.

- `routes/listingRoutes.js`  
  Listing routes.

- `routes/orderRoutes.js`  
  Order routes.

- `routes/otherRoutes.js`  
  Additional or miscellaneous routes not grouped elsewhere.

#### Rules for `routes/`

- Routes should define the URL structure and middleware chain.
- Routes should call controllers instead of implementing business logic inline.
- Preserve the existing file names and spelling, especially `favouriteRoutes.js`.
- When adding new endpoints, check whether they belong in an existing route file before creating a new one.

---

### `backend/uploads/`

- `uploads/`  
  Empty runtime directory for uploaded files or generated assets.

#### Rules for `uploads/`

- This folder is usually populated at runtime.
- Do not expect tracked source files here.
- If upload logic changes, make sure the middleware and Cloudinary/local handling still match the intended storage flow.

---

### `backend/utils/`

Utility helpers shared across the backend.

- `utils/emailService.js`  
  Email sending helper using Nodemailer.

- `utils/errorHandler.js`  
  Centralized error handling utilities.

#### Rules for `utils/`

- Put reusable non-route, non-controller logic here.
- Avoid duplicating email or error formatting logic in controllers.

---

### `backend/validators/`

Validators enforce request structure and data quality before business logic runs.

- `validators/authValidator.js`  
  Validation rules for auth requests.

- `validators/chatValidator.js`  
  Validation rules for chat requests.

- `validators/listingValidator.js`  
  Validation rules for listing requests.

- `validators/orderValidator.js`  
  Validation rules for order requests.

#### Rules for `validators/`

- Validate early, before controller logic runs.
- Keep validation rules aligned with the fields expected by the frontend and the controllers.
- If a route changes shape, update the matching validator at the same time.

4. `backend/routes/*.js` — see endpoint structure

6. `backend/controllers/*.js` — inspect business logic
7. `backend/middleware/*.js` — inspect shared request handling
8. `backend/config/schema.sql` — verify table structure
9. `backend/utils/*.js` — inspect reusable helpers
10. `backend/README.md` and `backend/api.testing.md` — confirm setup and usage expectations

## Backend conventions to preserve

- Use **CommonJS** (`require`, `module.exports`) unless a file is already clearly authored in TypeScript and intended for that workflow.
- Keep route files thin.
- Keep controllers focused on business logic.
- Keep validators close to the route data they protect.
- Keep error formatting centralized.
- Preserve the current folder names and filename spellings.
- Do not rename `favouriteRoutes.js` to `favoriteRoutes.js` unless all imports are updated.
- Treat `uploads/` as runtime-only.
- Keep SQL schema files in sync with DB setup scripts.
- Prefer extending existing modules over creating duplicate ones.

## Practical guidance for future edits

### If you are adding a new API feature

- Check whether there is already a matching route file.
- Add or update the validator first.
- Implement the controller logic next.
- Wire the route to the controller and middleware.
- Update schema files if the database changes.

### If you are changing authentication

- Inspect `authRoutes.js`, `authController.js`, `authMiddleware.js`, and `authValidator.js`.
- Verify token handling, password hashing, and protected-route logic together.

### If you are changing uploads

- Inspect `uploadMiddleware.js`, `emailService.js`, and any controller that stores image URLs.
- Verify whether the upload flow is local filesystem-based or Cloudinary-based.

### If you are changing the database

- Update `config/schema.sql`.
- Check `setup-db.js`.
- Compare against `config/original_schema.sql` if needed.
- Make sure the controllers and validators still match the data model.
